# 용량 최적화 검토 및 작업 플랜 (보훈 교재)

## 1. 현재 진단 (실측)
- 배포 총용량: 약 48MB
  - photos 35MB (98장, 평균 0.35MB)
  - assets 13MB (c1/bg 7.4 · c1/map 2.1 · c1/full 2.0 · 01_title 1.7)
- 포맷: 전부 PNG/JPG. WebP 미사용.
- 사진 과대해상도: 일부 원본 그대로. 예) 014-3.jpg = 12065x8842(106MP)/5.47MB 인데 실제 표시 폭은 ~600px.
- 공통 크롬 중복 baked: 모든 배경 PNG에 로고+레일6+상단4+그라데이션이 그대로 구워짐.
  두 페이지(read1 vs read2) 좌측 레일/상단 아이콘 영역 픽셀차 ≈ 0.2 → 사실상 동일.
  코스2~4까지 빌드하면 이 공통 픽셀이 페이지마다 ×4로 중복 누적.

## 2. 최적화 레버 (실측 절감)
A. 이미지 포맷/리사이즈 — 즉효·무위험·디자인 무변
   - 배경 PNG → WebP 무손실: 607KB→342KB(-44%), 카드 397→241KB
   - 사진 리사이즈(≤1600)+WebP q80: 5.47MB→128KB(-98%), 4.27MB→247KB, 454KB→101KB
   - 추정: photos 35MB→~4MB, assets 이미지 13MB→~7MB ⇒ 전체 48MB → 약 12~14MB
B. 공통 크롬 분리 (구조)
   - 그라데이션 배경 → CSS, 로고 → 공유 1장(또는 SVG), 레일/상단 아이콘 → SVG 세트
   - 배경에서 크롬 제거(이미 스켈레톤 추출 방식 보유) → 페이지 배경은 '콘텐츠 고유부'만
   - 효과: 중복 제거 + UI 선명(벡터) + 코스2~4 확장 시 중복 폭증 차단
C. 남은 UI/장식 코드화
   - 퀴즈 보기 박스·카드 테두리·구분선·말머리 등 코딩 가능한 요소 CSS/SVG로
D. 페이지타입 공통 템플릿 (4코스 공통)
   - reading / intro / quiz / map 프레임을 CSS 템플릿화, 배경은 콘텐츠 고유부만
E. 거버넌스
   - tools/ 에 이미지 최적화 빌드 스크립트 상시화 + "이미지 vs 코드" 판단 규칙 문서화
   - 코스2~4는 처음부터 이 규칙으로 가볍게 제작

## 3. 단계별 작업 플랜 (위험/효과 순)
[1단계] 이미지 파이프라인 최적화 — 0.5일, 무위험, 디자인 변화 없음
  - tools/optimize-images.py: 모든 배경 png→webp(무손실), photos 리사이즈+webp(q80)
  - 갤러리 확대용 사진은 2000px 보존 / 페이지 표시용은 1600px로 분리 가능
  - 로더(app.js)는 webp 우선 + png/jpg 폴백(<picture> 또는 onerror)
  - 기대: 48MB → 약 13MB
[2단계] 공통 크롬 레이어 — 1~1.5일, 중위험
  - 그라데이션 CSS화, 아이콘 SVG 세트, 배경 재추출 시 크롬 숨김
  - 페이지타입별 베이스 정리
[3단계] 페이지타입 템플릿 — 2~3일, 구조 개편
  - reading/intro/quiz 공통 프레임 CSS, 배경=콘텐츠 고유부만
  - 코스1에 적용·검증 → 코스2~4 확장 기준 확정
[4단계] 거버넌스 — 빌드 자동화 + 규칙 문서화

## 4. 리스크/주의
- WebP: 모든 현대 브라우저 지원(2020+). 안전하게 png/jpg 폴백 둘 것.
- 사료 사진 화질: 확대 팝업이 있으므로 갤러리 원본은 약간 큰 해상도 유지, 썸네일만 축소.
- 한글 파일명 + WebP: 문제 없음(.nojekyll 적용됨).
- photos.json/sitemap의 경로·확장자 참조 갱신 필요(스크립트가 함께 처리).
